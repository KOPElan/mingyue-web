# 功能规格：mingyue-web 可视化管理界面

**功能分支（Feature Branch）**: `001-mingyue-web-ui`  
**创建日期（Created）**: 2026-03-30  
**状态（Status）**: 草稿（Draft）  
**输入描述**: 本项目是 mingyue-go 的 Web 可视化管理界面，使用现代化前端技术将 mingyue-go 的能力转化为美观的 Web 界面，使用户可以在浏览器上管理 Linux 服务器。

## 澄清记录（Clarifications）

### Session 2026-03-30

- Q: API Key v1 阶段持久化策略是什么？ → A: localStorage + 首次"受信任设备"风险确认提示
- Q: 进程列表过滤方式是客户端还是服务端？ → A: 客户端过滤（一次性加载，前端 JS 实时过滤）
- Q: 文件上传机制与大小限制如何定义？ → A: 目标支持 1GB+ 大文件；但受限于 `POST /files` Base64 编码（1GB→1.33GB 内存占用），v1 不实现上传功能，仅保留占位 UI 并提示"上传功能将在后续版本支持（需 agent 侧 multipart 接口）"
- Q: 列表加载中与空数据状态如何统一展示？ → A: 骨架屏（Skeleton）加载 + 独立空状态图标+中文提示

## 用户故事与验收场景（User Scenarios & Testing）

### 用户故事 1 — Agent 连接配置与系统监控仪表板（优先级：P1）

运维人员打开浏览器，输入（或选择已保存的）mingyue-go agent 地址和 API Key，
成功连接后进入系统仪表板，实时查看 CPU 使用率、内存使用量、磁盘占用和系统运行时间，
并通过仪表板快速判断服务器当前是否正常。

**优先级理由**：这是所有其他功能的前提——如果无法连接 agent 或看不到基本状态，整个 Web UI 没有价值。

**独立可测试性**：仅凭本故事即可交付可演示的 MVP：配置页 + 仪表板页，用户完成配置后能看到实时系统数据。

**验收场景**：

1. **Given** 用户首次访问 Web UI，**When** 进入设置页并填写 agent 地址和 API Key，**Then** 系统在 3 秒内显示"连接成功"，并自动跳转到仪表板。
2. **Given** 用户已配置 agent，**When** 进入仪表板，**Then** 显示 CPU 使用率（百分比）、内存使用量（已用/总量）、系统运行时间，且数据每 5 秒自动刷新，不需要手动操作。
3. **Given** agent 不可达，**When** 用户尝试连接，**Then** 系统显示明确的中文错误提示（如"无法连接到 agent，请检查地址或服务状态"），不显示空白页或浏览器报错。
4. **Given** API Key 角色为 `viewer`，**When** 进入仪表板，**Then** 所有写操作入口（按钮/菜单）均显示为禁用状态，并提示"当前角色无权限执行此操作"。
5. **Given** 已切换至隐藏标签页，**When** 标签页不可见，**Then** 自动轮询暂停；重新激活标签页后轮询恢复，不产生冗余请求。

### 用户故事 2 — 进程管理（优先级：P2）

运维人员在进程管理页面查看服务器所有运行中的进程，通过搜索和过滤定位目标进程，
并在必要时通过确认对话框终止指定 PID 的进程。

**优先级理由**：进程管理是服务器日常运维的高频操作，影响面直接，优先级高于磁盘/文件操作。

**独立可测试性**：在仪表板（US1）完成后即可独立实现和测试：进程列表页 + 终止操作，无需其他功能。

**验收场景**：

1. **Given** 用户进入进程管理页，**When** 页面加载完成，**Then** 显示分页进程列表（包含 PID、进程名、状态、CPU%、内存 RSS、用户名）；首页默认显示 20 条，支持切换每页数量。
2. **Given** 进程列表已加载，**When** 用户在搜索框输入进程名关键词，**Then** 列表实时过滤，显示匹配的进程，无结果时显示"未找到匹配的进程"。
3. **Given** 用户点击某进程的"终止"按钮，**When** 角色为 `operator` 或 `admin`，**Then** 弹出确认对话框，显示"确认终止进程 [进程名]（PID: [pid]）？此操作不可撤销。"用户确认后执行终止，并显示操作结果。
4. **Given** 用户尝试终止进程，**When** 角色为 `viewer`，**Then** "终止"按钮为禁用状态，鼠标悬停时提示"需要 operator 或 admin 角色"。
5. **Given** 终止进程操作成功，**Then** 列表在 2 秒内自动刷新，已终止的进程从列表中消失。

---

### 用户故事 3 — 磁盘与挂载管理（优先级：P3）

运维人员在磁盘管理页面查看块设备列表和当前挂载点，
挂载本地磁盘、CIFS 网络共享或 NFS 共享，并在不需要时安全卸载挂载点；
同时通过 SMART 健康界面了解磁盘物理状态。

**优先级理由**：磁盘操作影响数据安全，属于重要管理操作，但频率低于进程管理。

**独立可测试性**：在 US1 基础上即可独立实现：设备列表 + 挂载表单 + 卸载操作 + SMART 展示。

**验收场景**：

1. **Given** 用户进入磁盘管理页，**When** 页面加载，**Then** 显示两个标签页——"块设备"（含设备名、大小、类型、型号）和"挂载点"（含设备、挂载路径、文件系统类型）。
2. **Given** 用户点击"新建挂载"，**When** 填写挂载类型（本地/CIFS/NFS）、源路径、目标路径并确认，**Then** 系统执行挂载请求；成功后列表更新，失败时显示明确的中文错误原因。
3. **Given** 挂载类型为 CIFS，**When** 用户填写用户名和密码，**Then** 密码字段为掩码显示；表单提交后组件状态中的凭据立即清除；浏览器控制台不输出任何凭据信息。
4. **Given** 用户点击某挂载点的"卸载"，**Then** 弹出确认对话框并显示挂载路径；确认后执行卸载，挂载路径在请求 URL 中被正确 URL 编码。
5. **Given** 用户点击设备的"SMART 健康"，**When** agent 返回健康数据，**Then** 显示健康状态（健康/警告/危险）、温度、通电时间等字段；若 agent 未安装 smartmontools，显示"设备未支持 SMART 查询"而非报错页。

### 用户故事 4 — 浏览器文件管理器（优先级：P4）

运维人员在文件管理器中以双栏（目录树 + 文件列表）方式浏览服务器文件系统，
支持预览文本文件、创建目录、移动/复制/删除操作，
全部通过 mingyue-go 文件 API 完成，无需 SSH；
文件上传功能因 API 层 Base64 编码限制（大文件内存膨胀问题），v1 阶段保留占位入口，延后实现。

**优先级理由**：文件管理是 Web UI 的核心差异化功能之一，但其复杂度较高，需要更多基础设施就绪后才能实现。

**独立可测试性**：单独的文件管理模块，可独立演示目录浏览和文件读取流程。

**验收场景**：

1. **Given** 用户进入文件管理器，**When** 页面加载，**Then** 左侧显示目录树（初始从 `/` 展开），右侧显示当前目录的文件列表（含名称、类型、大小、修改时间、权限）。
2. **Given** 用户点击一个文本文件，**When** 文件大小 ≤ 1MB，**Then** 右侧显示文件内容（经 `atob()` 解码后的文本），内容可滚动查看；文件大小 > 1MB 时提示"文件过大，仅支持下载"。
3. **Given** 用户编辑文件内容并点击"保存"，**When** 角色为 `operator` 或 `admin`，**Then** 内容经 `btoa()` 编码后通过 `POST /files` 写入，保存成功时显示"保存成功"提示。
4. **Given** 用户发起删除操作（文件或目录），**Then** 弹出确认对话框，显示"确认删除 [路径]？目录删除将递归删除所有内容，不可恢复。"确认后执行删除。
5. **Given** 用户拖动文件到另一目录，**When** 目标目录不同于源目录，**Then** 触发移动操作（`PUT /files/move`），操作完成后两侧目录树和列表自动刷新。

---

### 用户故事 5 — 共享管理（Samba/NFS）（优先级：P5）

运维人员在共享管理页面查看、创建、修改和删除 Samba 共享与 NFS 导出，
并管理 Samba 用户账号，无需直接编辑配置文件。

**优先级理由**：共享管理是 NAS 场景的核心功能，但受众相对具体，优先级低于基础监控和文件管理。

**独立可测试性**：独立的共享管理模块，可在无其他功能的情况下完整验证共享 CRUD 流程。

**验收场景**：

1. **Given** 用户进入共享管理页，**When** 切换"Samba"和"NFS"标签，**Then** 分别显示对应共享列表，每项显示名称、路径、启用状态、只读/读写标记。
2. **Given** 用户点击"创建 Samba 共享"，**When** 填写名称和路径并确认，**Then** 系统调用 `POST /smb/shares` 创建共享；成功后列表更新；名称冲突时显示"共享名称已存在（CONFLICT）"。
3. **Given** 用户点击某共享的"删除"，**Then** 弹出确认对话框，显示被删除共享的名称和路径；删除后列表刷新。
4. **Given** 用户进入"Samba 用户"子页，**When** 创建新用户，**Then** 密码字段为掩码输入，用户创建后不在界面上显示密码；修改密码时同样如此。

---

### 用户故事 6 — 网络接口管理与 ACL 管理（优先级：P6）

管理员（`admin` 角色）查看服务器网络接口状态，并可执行启用/禁用/刷新 DHCP 操作；
同时可以查询和设置文件或目录的 POSIX ACL 权限条目。

**优先级理由**：网络变更风险高（误操作可导致服务器失联），仅 admin 可用，优先级相对靠后；ACL 管理频率较低。

**独立可测试性**：独立的网络管理和 ACL 两个子功能，可分别独立测试。

**验收场景**：

1. **Given** 用户进入网络管理页，**When** 页面加载，**Then** 显示所有网络接口卡片，每张卡片显示接口名、MAC、IP 地址列表、MTU、运行状态（up/down）。
2. **Given** 用户角色为 `admin` 并点击接口的"启用/禁用"，**Then** 弹出确认对话框，明确说明"[操作] 接口 [接口名] 可能导致当前连接中断"；确认后执行操作，结果反馈显示中文提示。
3. **Given** 用户角色为 `viewer` 或 `operator`，**When** 进入网络管理页，**Then** up/down/dhcp 操作按钮全部禁用，鼠标悬停显示"需要 admin 角色"。
4. **Given** 用户在 ACL 管理页输入路径并查询，**When** 路径有效，**Then** 显示文件所有者、组、UNIX 权限位（如 `0755`）以及 POSIX ACL 条目列表；若 `getfacl` 未安装，ACL 条目显示空列表并附注说明。
5. **Given** 用户添加 ACL 条目（如 `u:alice:rwx`），**When** 角色为 `operator` 或 `admin` 并确认，**Then** 调用 `PUT /acl`；条目格式不合法时（不符合 `[u|g|o|m]:[qualifier]:[rwx-]`）在提交前完成前端校验并提示格式错误。

### 用户故事 7 — 多 Agent 管理（优先级：P7）

用户在设置页管理多台宿主机 agent（增删改查 agent 列表），
在应用顶部通过切换器快速在不同宿主机之间切换，
无需重新打开浏览器或更改 URL。

**优先级理由**：单 agent 场景先跑通后再扩展多 agent，属于增强功能。

**独立可测试性**：在基础 US1 完成后可独立实现 agent 切换器，而无需其他高级功能。

**验收场景**：

1. **Given** 用户在设置页，**When** 点击"添加 Agent"，**Then** 弹出表单填写主机名标签、agent 地址、API Key；保存后新 agent 出现在列表中，并通过 `GET /health` 验证连通性，显示"在线"或"离线"状态。
2. **Given** 已添加多台 agent，**When** 用户在顶部切换器选择不同 agent，**Then** 全局 API 上下文切换为对应 agent 的地址和 Key，所有数据请求立即指向新 agent；切换后仪表板数据自动刷新。
3. **Given** 某个 agent 离线，**When** 用户尝试切换到该 agent，**Then** 所有数据区域显示"agent 不可达"的占位状态，不显示上一个 agent 的残留数据。

### 边界情况（Edge Cases）

- **agent 版本不一致**：当 agent `/version` 返回的版本低于已知要求时，显示版本警告提示，但不阻止使用。  
- **路径越权**：文件管理器中路径穿越操作（如输入 `../../etc/passwd`）由 agent 侧防护，前端收到 `403 FORBIDDEN` 时显示"路径访问被拒绝（安全限制）"。  
- **大文件读取**：文件大小超过 1MB 时禁用浏览器端预览，提供直接下载按钮，防止 Base64 解码导致 OOM。  
- **并发切换**：用户在上一次 API 请求返回前切换 agent，前一请求结果 MUST 被丢弃，不更新当前 agent 的 UI 状态。  
- **网络中断恢复**：浏览器网络恢复后，健康检查轮询自动重连，不需要用户手动刷新页面。  
- **CIFS 挂载失败**：若 CIFS 凭据错误，显示"挂载失败：认证错误"，不在提示中暴露原始凭据。

## 功能需求（Requirements）

### 核心功能需求（Functional Requirements）

**连接与认证**
- **FR-001**：系统 MUST 支持用户配置 mingyue-go agent 的地址（URL）和 API Key，配置通过 `localStorage` 持久保存（跨会话保留）。
- **FR-001a**：首次在当前浏览器保存 API Key 前，系统 MUST 弹出"受信任设备确认"提示，告知用户 Key 将存储在本地浏览器存储中并提示仅在受信任设备上使用；用户确认后方可写入 localStorage。
- **FR-002**：系统 MUST 通过 Bearer Token 方式向 agent 发起所有 API 请求（健康检查除外）。
- **FR-003**：系统 MUST 识别 `401 UNAUTHORIZED` 响应并引导用户重新配置 API Key，不静默失败。
- **FR-004**：系统 MUST 识别 `403 FORBIDDEN` 响应并向用户展示中文权限不足提示（含所需角色信息）。

**系统监控**
- **FR-005**：系统 MUST 展示 CPU 使用率、内存总量/已用量/使用率、系统运行时间，数据来源为 `GET /system/overview`。
- **FR-006**：系统 MUST 以不超过 5 秒的频率自动刷新仪表板数据；在浏览器标签不可见时暂停刷新。

**进程管理**
- **FR-007**：系统 MUST 一次性加载所有进程并展示分页列表；关键词过滤 MUST 在客户端 JS 中实时执行（不触发额外 API 请求），无结果时显示"未找到匹配的进程"。
- **FR-008**：系统 MUST 在执行终止进程前弹出确认对话框，包含明确的中文操作描述。
- **FR-009**：终止进程操作 MUST 仅在当前角色为 `operator` 或 `admin` 时可操作；`viewer` 角色时按钮禁用。

**磁盘管理**
- **FR-010**：系统 MUST 支持本地磁盘、CIFS、NFS 三种挂载类型的挂载操作。
- **FR-011**：CIFS 挂载时密码字段 MUST 为掩码输入，提交后组件状态立即清除凭据，不允许持久化。
- **FR-012**：卸载操作的请求 URL MUST 对挂载路径进行 `encodeURIComponent()` 编码。
- **FR-013**：系统 MUST 展示设备 SMART 健康信息；未安装 `smartmontools` 时显示降级提示而非报错。

**文件管理**
- **FR-014**：系统 MUST 提供目录树 + 文件列表的双栏布局，支持目录导航。
- **FR-015**：读取文件内容时 MUST 用 `atob()` 解码 Base64；写入文件时 MUST 用 `btoa()` 编码。
- **FR-016**：文件大小 > 1MB 时 MUST 禁用浏览器预览，提供下载入口。
- **FR-016a**：v1 阶段不实现文件上传功能；文件管理器工具栏中 MUST 保留"上传文件"占位按钮（禁用状态），鼠标悬停时显示提示"上传功能将在后续版本支持（需服务端 multipart 接口）"；目标支持 1GB+ 文件上传，待 mingyue-go 提供 multipart/form-data 接口后实现。
- **FR-017**：删除操作（含目录）MUST 弹出确认对话框，告知递归删除不可恢复。

**共享管理**
- **FR-018**：系统 MUST 分别展示 Samba 共享和 NFS 导出列表，支持各自的创建、修改、删除操作。
- **FR-019**：系统 MUST 提供 Samba 用户管理（创建/修改密码/删除），密码全程掩码，不回显。

**网络与 ACL**
- **FR-020**：网络接口的 up/down/dhcp 操作 MUST 仅在 `admin` 角色时可操作，其他角色按钮禁用。
- **FR-021**：执行网络接口变更前 MUST 弹出确认对话框，告知可能导致连接中断的风险。
- **FR-022**：ACL 条目输入格式 MUST 在前端进行格式校验（正则：`^[ugom]:[^:]*:[rwx-]{3}$`），格式错误时提交前显示错误提示。

**交互与 UX 流程**
- **FR-025**：所有列表类组件（进程列表、挂载点列表、文件列表、共享列表、接口列表等）在数据请求期间 MUST 显示骨架屏（Skeleton）占位，不显示空白或 spinner。
- **FR-026**：所有列表类组件在数据加载完成且结果为空时，MUST 显示专属空状态区域，包含图标和中文说明文字（如"暂无进程数据"、"目录为空"），明确区别于"正在加载"状态。
- **FR-023**：系统 MUST 支持管理多个 agent 配置（增删改），并在切换 agent 时完全隔离 API 上下文。
- **FR-024**：系统 MUST 通过 `GET /health` 定期检测每个 agent 的在线状态并显示在 agent 列表中。

### 关键实体（Key Entities）

- **Agent 配置（AgentConfig）**：代表一台 mingyue-go 宿主机的连接信息，包含：用户自定义标签、agent 地址、API Key（安全存储，不进 localStorage）、角色（从 API 响应推断或用户声明）、在线状态。
- **进程（Process）**：PID、进程名、状态、CPU 百分比、内存 RSS、所属用户、命令行、（可选）可终止标志。
- **挂载点（Mount）**：设备路径、挂载目标路径、文件系统类型、挂载选项、挂载类型（local/cifs/nfs）。
- **文件条目（FileEntry）**：名称、类型（file/dir/symlink）、大小、修改时间、权限位、路径。
- **共享（Share）**：名称、类型（smb/nfs）、路径、只读标志、启用状态、访问限制配置。
- **网络接口（NetworkInterface）**：接口名、MAC、IP 地址列表、MTU、运行状态、标志位列表。
- **ACL 条目（AclEntry）**：标签（user/group/other/mask）、限定符、权限字符串（rwx-）。

## 成功标准（Success Criteria）

### 可量化指标（Measurable Outcomes）

- **SC-001**：首次连接配置时间 — 用户从打开浏览器到成功看到仪表板数据，全程操作 MUST ≤ 2 分钟。
- **SC-002**：首次内容渲染（FCP） — 仪表板页面在局域网正常连通条件下 MUST ≤ 1.5 秒。
- **SC-003**：危险操作防误触率 — 所有破坏性操作（终止进程、卸载磁盘、删除文件、down 网络接口）100% 经过确认对话框，0 例可以一步操作直接执行。
- **SC-004**：权限一致性 — `viewer` 角色进入任何页面时，所有写操作入口 100% 处于禁用或不可见状态，无需依赖 agent 侧 403 来防止越权。
- **SC-005**：错误可读性 — 所有从 agent 返回的错误码（UNAUTHORIZED/FORBIDDEN/NOT_FOUND/CONFLICT/INTERNAL）100% 转换为中文用户提示，不出现原始英文错误码。
- **SC-006**：初始加载包体积 — gzip 压缩后的初始 JavaScript Bundle MUST ≤ 300KB，通过路由级代码分割实现按需加载。
- **SC-007**：CIFS 凭据安全 — CIFS 密码在浏览器 devtools（Network 标签、Console、Application/Storage）中 MUST 不可见，通过安全审查验证。
- **SC-008**：多 Agent 切换 — 在已添加 3 台 agent 的情况下，切换 agent 后新 agent 数据完全显示（含仪表板刷新） MUST ≤ 3 秒。

## 假设与依赖（Assumptions）

- **前提依赖**：目标宿主机上已运行 mingyue-go agent（版本 ≥ Phase 7，支持 API Key 持久化和 LAN 发现）；agent 端口 7070 在网络上可达。
- **部署模式**：Web UI 运行在局域网内的浏览器（或通过 nginx 反代访问），不要求公网部署；CORS 问题由 nginx 反代层处理，agent 本身不启用 CORS。
- **用户群体**：目标用户为具备基本 Linux 知识的运维人员或 NAS 爱好者，无需向非技术用户解释概念，但 UI 文案 MUST 使用中文且避免裸露的系统术语。
- **移动端**：桌面浏览器（≥ 1024px）为 v1 目标，移动端适配为未来增强，当前规格不约束。
- **API Key 存储方案**：v1 阶段采用 `localStorage` 持久存储 API Key，实现跨会话免重复输入，适合家庭 NAS 等受信任设备场景。首次保存时 MUST 展示"受信任设备确认"对话框（见 FR-001a）；NAS 家庭用户在自有设备上使用时便捷性优先，风险由用户知情确认承担。后端代理方案作为未来安全增强路径，不纳入 v1 范围。
- **日志与审计**：前端不实现审计日志，审计由 agent 侧负责；前端仅保证不把 key 和密码输出到 `console`。
- **国际化（i18n）**：v1 阶段仅支持简体中文界面，不实现 i18n 框架，硬编码中文文案可接受。
